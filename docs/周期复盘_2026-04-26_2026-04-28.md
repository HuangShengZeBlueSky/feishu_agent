## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

这周主要搞定的是“把演示链路变成真实链路”。

具体包括：

- 新建真实飞书多维表格，并创建 4 张核心表：`Project_Context_Sources`（上下文来源表）、`Project_Action_Items`（行动项表）、`Project_Decisions`（决策表）、`Project_Risks`（风险表）。
- 写入最小真实 seed 数据。`seed` 的中文含义是“种子数据”：它不是 mock 文件，而是写进真实飞书 Base 的最小项目数据，用来保证系统没有真实业务资料时也能跑通。
- 新增 LLM JSON 抽取层，用 Ark/OpenAI 兼容接口让模型输出行动项、决策、风险等结构化字段。
- 把 CLI 主入口切到真实 Feishu + LLM，包括上下文概览、会后闭环、会前卡片、推进表对账、风险周报、会议排期依赖、文档预览。
- 接入 OpenClaw 飞书通道，并把飞书客户端里的机器人显示名对齐为 `会议`。
- 建立 `validate-xiaolongxia` 验收命令，把真实业务链路、OpenClaw 飞书通道、lark-cli 与 OAuth 状态放到同一个验收结果里。

AI 的分工主要是：辅助拆分真实链路改造步骤、生成结构化抽取提示词、设计验收维度、整理文档和测试命令。人工主要负责确认真实飞书资源、授权 OAuth、判断哪些写入动作可以执行。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

最明显的卡点是 `run-all-real --dry-run` 一次串行跑多个 LLM 步骤时超过 6 分钟命令超时。这个问题不是某一个功能一定失败，而是多个 LLM 调用串在一起后耗时太长。

破局方式是改成单命令验收：

- 单独跑 `context-overview` 看真实 Base 能不能读。
- 单独跑 `post-meeting-real --dry-run` 看 LLM 抽取和写入预览是否正常。
- 单独跑会前卡片、推进表对账、风险周报、排期依赖、文档预览。
- 最后用 `validate-xiaolongxia` 做总体验收。

第二个卡点是权限边界。个人 OAuth 可以访问用户个人资源，但不能假设永久有效；bot 身份适合长期运行，但看不到用户个人日历、个人妙记等资源。最后采用的原则是：能用 bot 的长期能力尽量用 bot；只有读取个人资源时才用 user OAuth，并在缺权限时明确说明缺什么。

### 3. 有没有什么这次写得特别顺，或者下次还能直接复用的东西？

可以直接复用的沉淀：

- 真实资源缺口模板：`缺什么 -> 从哪里来 -> 谁能提供 -> 怎么验证 -> 卡住哪个能力`。
- `validate-xiaolongxia` 验收思路：一条命令同时检查真实业务链路、OpenClaw 飞书通道、lark-cli 与 OAuth。
- LLM JSON 抽取套路：明确要求模型只根据真实数据输出 JSON，并保留 `source_quote`（来源原文证据）。
- A/B/D 路由结构：A 处理风险和周报，B 处理会议前后链路，D 处理待办中枢和推进表对账。

最值得保留的工程原则是：缺真实数据时可以用真实 Base 里的 seed 数据兜底，但不能回退到本地 mock；缺用户、日历、妙记等信息时明确说缺口，不编造。

## 四、本周随手记（选填）

- 个人 OAuth 不能真正永久授权。`offline_access` 可以让 token 在一段时间内刷新，但 refresh token 过期后仍然需要重新授权。
- 长期运行能力优先走 bot 身份；个人日历、个人妙记、个人云空间这类用户私有资源才走 user OAuth。
- OpenClaw 能力探测里 probe 偶尔会 10 秒超时，但只要 Feishu main 显示 running、send/read 等能力存在、群白名单正确，仍需要用飞书客户端实际 @ 一次做端到端确认。
- 这次最大的收敛不是多做一个功能，而是把“数据从哪里来、能不能验证、缺什么不编造”这套规则定下来了。
