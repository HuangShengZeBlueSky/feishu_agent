#### 3）项目亮点介绍

项目亮点 1：围绕会议场景做主动性 AI，而不是普通聊天机器人。

小龙虾不是等用户问一句再答一句，而是围绕项目会议生命周期主动工作：会前准备、会后闭环、周期风险巡检、推进表对账和文档沉淀。

项目亮点 2：真实飞书资源闭环。

项目主路径已经从 mock 数据切到真实飞书 Base、真实文档、真实任务、真实日程和真实飞书入口，输出结果能落到飞书工作流里。

项目亮点 3：A/B/D 三条主线清晰。

- A：周期性总结与风险洞察，面向周报和风险巡检。
- B：会议与项目全链路伴侣，面向会前准备、会后闭环和排期依赖。
- D：团队待办中枢与推进表对账，面向任务同步、重复合并和人工确认。

项目亮点 4：可控的 AI 写入策略。

推进表对账不是让 AI 直接乱改项目表，而是先输出 `diff`（差异清单），把新增、更新、合并、关闭、需要人工确认拆开，再由人确认。

项目亮点 5：可验证 Demo。

项目有 `run_logs` 运行日志和 `validate-xiaolongxia` 验收器，当前真实链路验收 17/17 通过。

#### 4）AI 亮点介绍

高阶 AI 技巧 1：主动性 AI 触发信号设计。

项目把主动性拆成可判断的信号：延期、阻塞、缺负责人、缺截止时间、重复任务、字段冲突、未闭环决策。AI 根据这些信号主动生成风险提醒、会议材料和对账建议。

高阶 AI 技巧 2：LLM JSON 结构化输出。

模型不是只写自然语言总结，而是输出可被程序继续处理的 JSON 字段，例如行动项、决策、风险、今日风险卡片、下周关注、推进表 diff。

高阶 AI 技巧 3：证据保留。

行动项、决策和风险都保留 `source_quote`（来源原文证据），让用户知道 AI 结论来自哪句话，降低幻觉风险。

高阶 AI 技巧 4：人机协同。

AI 负责读上下文、抽取结构、识别风险、生成建议；人负责确认冲突项、决定是否写回、承担最终业务责任。

高阶 AI 技巧 5：模型和工具解耦。

LLM 客户端采用 OpenAI 兼容接口，可以接入豆包 Seed 2.0 Pro 对应的火山 Ark endpoint；如果模型不支持严格 JSON 模式，也会降级为普通 chat 后解析 JSON。

引入 AI 后的工作流变化：

```text
原来：
项目经理人工翻飞书群、妙记、任务、文档 -> 手写会前材料 -> 会后手工整理行动项 -> 每周手写风险周报

现在：
小龙虾读取真实项目上下文 -> AI 生成会前卡片、行动项、风险周报、推进表 diff -> 人确认关键变更 -> 结果沉淀回飞书
```

## 三、其他信息（可选）- 自由发挥区

本项目最核心的思路是：主动性 AI 不应该只是“定时总结”，而应该知道自己为什么介入、什么时候介入、介入后输出什么、哪些事情必须交给人确认。

小龙虾在会议场景里的定位是：

```text
看见上下文 -> 判断风险和缺口 -> 主动生成建议 -> 人确认 -> 沉淀回飞书
```

这个思路既适合复赛 Demo，也适合后续继续扩展为真实项目协作助手。
