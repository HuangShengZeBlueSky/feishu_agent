# 小龙虾项目上下文管家：单一审查计划与缺口清单

更新时间：2026-04-25

## 1. 一句话版本

先只验收一条链路：

```text
本地会议文本
  -> 抽取行动项 / 决策 / 风险
  -> 输出 JSON 和会后摘要
  -> dry-run 预览写入飞书多维表格
  -> 配齐表 ID 后再真实写入
```

这一版不做全飞书搜索、不做 Web 前端、不做自动改文档、不做 WebSocket 事件。先让你能审查“抽取对不对、字段对不对、写入前会写什么”。

## 1.1 六个例子收敛成一个验收主线

你根目录的 `测试六个例子,没有真人参与如何测试.md` 提到六类黑盒验证：会后行动项、会前背景卡片、项目推进表、延期提醒、自动约会/查忙闲、文档生成更新。为了避免第一版摊太大，当前统一收敛为一个主线：

```text
mock 模式先跑通本地数据
  -> 多维表格作为唯一结构化落点
  -> 所有真实写操作默认 dry-run
  -> 再逐个打开消息、文档、任务、日历、会议妙记
```

这意味着你审查时不用看代码，只看四类黑盒证据：

| 证据 | 中文含义 | 通过标准 |
|---|---|---|
| 命令输出 | CLI 是否成功 | 返回 `ok=true` 且 counts 合理 |
| JSON 文件 | 结构化抽取结果 | 行动项、决策、风险分开，不混写 |
| 写入预览 | 真实写入前的 dry-run 清单 | 能看到将写入哪些字段 |
| 飞书产物 | 多维表格/群消息/文档/任务 | 只在你显式开启写入后产生 |

## 3. 审查步骤

### Step 1：本地抽取是否可接受

运行：

```powershell
python -m src.cli post-meeting-local --minutes-file resources\mock\sample_minutes.txt
```

你只需要审查四个文件：

```text
output/action_items.json
output/decisions.json
output/risks.json
output/post_meeting_summary.md
```

审查重点：

| 审查项 | 中文含义 | 为什么重要 |
|---|---|---|
| `title` | 行动项标题 | 决定多维表格里是否好扫读 |
| `owner` | 负责人 | 决定能不能创建飞书任务 |
| `owner_text` | 负责人文本 | 没有真实飞书用户时只填这个，不编造 user ID |
| `owner_user_id` | 飞书用户 ID | 只有确认真实 `open_id` / `user_id` 后才填写 |
| `due_date` | 截止时间 | 决定每日风险巡检能不能判断延期 |
| `source_quote` | 原文证据 | 用来降低幻觉，每条结论都应有来源 |
| `confidence` | 抽取置信度 | 后续决定哪些事项可以自动创建任务 |

### Step 2：检查多维表格写入预览

运行：

```powershell
python -m src.cli post-meeting-local `
  --minutes-file resources\mock\sample_minutes.txt `
  --write-bitable `
  --dry-run
```

只看：

```text
output/bitable_write_preview.json
```

这个文件就是“如果真的写入飞书，会写哪些记录”。你确认字段没问题后，再配置真实表 ID。

### Step 3：配置真实多维表格并写入

需要先准备一个飞书多维表格，里面至少三张表：

```text
Project_Action_Items
Project_Decisions
Project_Risks
```

然后配置：

```powershell
$env:FEISHU_BITABLE_APP_TOKEN="..."
$env:FEISHU_ACTION_TABLE_ID="..."
$env:FEISHU_DECISION_TABLE_ID="..."
$env:FEISHU_RISK_TABLE_ID="..."
```

真实写入命令：

```powershell
python -m src.cli post-meeting-local `
  --minutes-file resources\mock\sample_minutes.txt `
  --write-bitable
```

## 4. 统一缺口清单

详细版见：

```text
docs/飞书依赖统一清单.md
```

以后任何缺口都按统一格式描述：

```text
缺什么 -> 从哪里来 -> 谁能提供 -> 怎么验证 -> 卡住哪个能力
```

### 4.1 基础凭证

| 缺口 | 当前状态 | 需要你提供/确认 |
|---|---:|---|
| `FEISHU_APP_ID`，飞书应用 ID | 已存在 | 不需要重复给我 |
| `FEISHU_APP_SECRET`，飞书应用密钥 | 已存在 | 不需要重复给我 |
| `tenant_access_token`，租户访问令牌 | 可自动获取 | 不需要手动提供，会由脚本换取 |
| OpenClaw Gateway | 未运行 | 建议后续装 WSL2 后再跑 |
| OpenClaw Feishu plugin | Windows 原生加载失败 | 先不用它做 MVP，直接用 Feishu OpenAPI |

### 4.2 多维表格

| 缺口 | 用途 | 当前状态 |
|---|---|---:|
| `FEISHU_BITABLE_APP_TOKEN` | 指向项目总多维表格 | 缺 |
| `FEISHU_ACTION_TABLE_ID` | 行动项写入表 | 缺 |
| `FEISHU_DECISION_TABLE_ID` | 决策写入表 | 缺 |
| `FEISHU_RISK_TABLE_ID` | 风险写入表 | 缺 |
| 多维表格写权限 | 创建/写入记录 | 已实测当前 App 具备创建 app/table/record/update 的能力 |

最低可用字段可以先用文本字段承接，后续再把 `owner` 改成人员字段、`status` 改成单选字段、`due_date` 改成日期字段。

### 4.3 群聊与卡片

| 缺口 | 用途 | 建议权限/对象 |
|---|---|---|
| `FEISHU_TEST_CHAT_ID` 或 `FEISHU_CHAT_ID` | 发送会后摘要、风险提醒、确认卡片 | 缺 |
| 机器人加入目标群 | 机器人不在群里就无法发群消息 | 缺确认 |
| `im:message:send_as_bot` | 以机器人身份发消息/卡片 | 需要在飞书开放平台确认 |
| `im:message` | 消息读写基础权限 | 需要在飞书开放平台确认 |
| `im:message.group_at_msg:readonly` | 读取群里 @ 机器人的消息 | 后续做群指令时需要 |
| `im:resource` | 上传图片/文件资源 | 后续发图片/附件时需要，纯文本/普通卡片可先不依赖 |

当前脚本已支持三类卡片的探测：会前卡片、会后卡片、风险卡片。但为了避免误发，必须同时设置：

```powershell
$env:FEISHU_ENABLE_WRITES="1"
$env:FEISHU_ENABLE_CARD_PUSH="1"
```

### 4.4 文档

| 缺口 | 用途 | 建议权限/对象 |
|---|---|---|
| `FEISHU_TEST_DOC_ID` 或 `FEISHU_DOC_ID` | 读取项目背景文档 | 缺 |
| 文档本身授权给机器人或应用 | tenant token 也要能访问该文档 | 缺确认 |
| 新版文档读取权限 | 读取 docx 原文/块内容 | 需要在飞书开放平台确认“查看新版文档”类权限 |

当前策略：第一版不自动修改文档，只读项目上下文清单里登记过的文档。

### 4.5 会议妙记

| 缺口 | 用途 | 当前 API 探针返回 |
|---|---|---|
| `FEISHU_TEST_MINUTE_TOKEN` 或 `FEISHU_MEETING_MINUTE_TOKEN` | 指向具体会议妙记 | 缺 |
| `minutes:minutes.basic:read` | 读取妙记基础信息/摘要类信息 | 当前缺 |
| `minutes:minutes:readonly` 或 `minutes:minutes` | 读取妙记内容 | 当前缺 |
| `minutes:minutes.transcript:export` | 导出逐字稿 | 当前缺 |
| `minutes:minute:download` | 下载妙记相关内容 | 当前缺 |
| 用户身份授权 | 读取用户可见的妙记 | 很可能需要，机器人 tenant token 不一定够 |

当前结论：会议妙记是第一版最关键但还没打通的源。没有真实 `minute_token` 和权限前，先用本地会议文本验收抽取链路。

### 4.6 任务

| 缺口 | 用途 | 当前 API 探针返回 |
|---|---|---|
| `FEISHU_TEST_TASK_GUID` 或 `FEISHU_TASK_GUID` | 读取/同步具体任务 | 缺 |
| `task:task:read` | 读取任务 | 当前缺 |
| `task:task:write` | 创建/更新任务 | 当前缺 |

安全规则：只有 `confidence >= 0.8`、有负责人、有截止时间的行动项，后续才允许自动创建任务；默认仍然 dry-run。

### 4.7 日历

| 缺口 | 用途 | 当前 API 探针返回 |
|---|---|---|
| `FEISHU_TEST_CALENDAR_ID` 或默认日历访问 | 会前提醒、会前卡片 | 缺 |
| `calendar:calendar:readonly` | 读取日历 | 当前缺 |
| `calendar:calendar:read` | 读取日历 | 当前缺 |
| `calendar:calendar.calendar:readonly` | 读取日历相关资源 | 当前缺 |
| `calendar:calendar` | 日历读写总权限 | 可选，MVP 只读优先 |

当前结论：日历不是阶段 1/2 的 blocker，等会后链路稳定后再接会前触发。

## 5. 本阶段不需要的东西

| 项目 | 为什么先不需要 |
|---|---|
| WebSocket 事件订阅 | MVP 先用手动命令和定时任务，稳定后再接事件 |
| 文档写入权限 | 第一版不自动改文档 |
| 删除权限 | 明确禁止 |
| 全公司搜索权限 | 与项目上下文清单原则冲突 |
| 多项目权限系统 | 先只做一个项目群 |
