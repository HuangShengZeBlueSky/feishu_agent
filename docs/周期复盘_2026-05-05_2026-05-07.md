### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

第一个卡点是：复赛材料不能只写“我接了飞书 API 和大模型”，否则会显得像普通工具拼接，价值不够清楚。

破局方式是把表达顺序改成：

```text
痛点 -> 主动性 AI 介入方式 -> A/B/D 闭环 -> Demo 证据 -> 技术架构
```

这样评委先看到为什么需要这个项目，再看到 AI 到底主动做了什么，最后再看技术实现。

第二个卡点是：完整链路包含飞书、Base、LLM、文档、日程、任务和 OpenClaw，直接录一条超长链路不稳定。

破局方式是把 Demo 拆成短步骤，每一步都能单独解释价值：

- 项目状态查询：证明系统读到了真实上下文。
- 会前卡片：证明 AI 能主动准备会议。
- 风险周报：证明 AI 能主动发现问题。
- 推进表对账：证明 AI 能主动指出差异，但不越权改表。
- 文档沉淀：证明结果能进入项目知识库。

第三个卡点是：主动性 AI 容易被质疑“会不会乱改东西”。破局方式是强调安全边界：AI 先输出 `diff` 和 `source_quote`，把新增、更新、合并、关闭、需要人工确认拆开，人确认后再写回。

### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

这次最顺的是“复赛三维度对齐模板”，后续答辩、README、PPT 都可以复用：

## 四、项目介绍 + Demo 录屏演示 + 项目链接

### 项目简介

小龙虾项目上下文管家是一个面向飞书会议和项目协作的主动性 AI 助手。它读取真实飞书项目数据，用 LLM 做结构化抽取、风险识别和文档生成，围绕会前准备、会后闭环、周期风险洞察和推进表对账形成完整工作流。

它解决的核心问题是：项目会议后，行动项、风险、决策和文档经常分散在群聊、妙记、任务和表格里，人工整理慢、容易漏、难追踪。小龙虾把这些信息统一到真实飞书事实库里，再由 AI 主动生成可执行结果。
