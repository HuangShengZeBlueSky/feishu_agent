# 第四周期（2026-05-02 ~ 2026-05-04）

## 文档摘要

这个周期的核心成果是：围绕“主动性 AI + 会议场景”，把第三周期设计的 A/B/D 三条主动链路推进成可演示闭环。

如果说第三周期解决的是“主动性会议 AI 应该怎么设计”，那第四周期解决的是“它如何真的在项目会议流程里发挥作用”。本周期最终实现了三个方向：

- A：周期性总结与风险洞察，面向周报和风险巡检。
- B：会议与项目全链路伴侣，面向会前准备、会后闭环和排期依赖。
- D：团队待办中枢与推进表对账，面向任务同步、重复合并和人工确认。

这三个方向共同构成小龙虾的主动性 AI 闭环：它会持续读取飞书项目上下文，主动发现风险和待办问题，主动生成会议材料和周报洞察，主动指出推进表差异，再把结果沉淀回飞书文档、Base 或任务流。

## 一、核心产出（必填）

本周期最满意的成果是：完成“小龙虾主动性会议 AI”的 A/B/D 三条产品主链路，并形成复赛可演示版本。

### A：周期性总结与风险洞察

A 方向解决的问题是：项目风险通常不是突然出现的，而是从“任务快到期、负责人缺失、环境阻塞、决策未落地”慢慢积累出来的。人工周报很容易只写进度，不写真实风险。

本周期实现后，小龙虾可以主动巡检真实飞书 Base：

- 扫描未完成行动项，识别高优先级任务。
- 检查缺负责人、缺截止时间、阻塞原因和延期风险。
- 汇总今日风险卡片、本周风险洞察、下周关注重点和需要管理者关注的事项。
- 输出适合直接放进周报的结构化内容。

AI 的关键作用是：它不是等用户问“有什么风险”，而是根据任务状态和项目上下文主动把风险提出来。

### B：会议与项目全链路伴侣

B 方向解决的问题是：会议通常不是单点事件。真正低效的地方在于会前没人整理背景、会中问题散落、会后行动项没有闭环、下一次会议又重新问一遍。

本周期实现后，小龙虾覆盖了会前、会后和排期依赖：

- 会前：主动生成背景卡片，列出历史决策、未完成事项、当前风险和建议追问。
- 会后：从会议文本中抽取行动项、决策和风险，并保留 `source_quote` 作为来源证据。
- 排期：当缺少参会人 open_id、日历授权或会议室信息时，主动列出依赖缺口，不编造候选时间。
- 沉淀：生成项目文档预览或真实飞书文档，让会议结论变成可复用资料。

AI 的关键作用是：它把会议从“聊完就散”变成“会前准备、会后跟进、下次复用”的连续流程。

### D：团队待办中枢与推进表对账

D 方向解决的问题是：项目推进表经常和会议纪要、群聊任务、飞书任务不同步。重复任务、口径冲突、缺负责人和未关闭事项会让团队执行混乱。

本周期实现后，小龙虾可以主动生成推进表 `diff`（差异清单）：

- 新增：会议里出现但推进表里没有的任务、风险或决策。
- 更新：已有任务状态、负责人、优先级或风险等级发生变化。
- 合并：同一事项被重复记录时，提示合并。
- 关闭：已经完成或失效的事项，提示关闭。
- 需要人工确认：AI 无法安全判断的冲突项，交给人确认。

AI 的关键作用是：它不直接越权改表，而是先主动发现差异，再让人确认。这既体现主动性，也保证工程和业务上的可控性。

## 二、量化指标（必填）

- 主动性 AI 闭环：3 条主链路，分别覆盖 A 风险洞察、B 会议全链路、D 待办对账。
- 会议场景覆盖：会前准备、会后闭环、风险巡检、排期依赖、文档沉淀、推进表对账。
- AI 结构化产物：会前卡片、行动项、决策、风险、风险周报、推进表 diff、飞书文档预览。
- 真实数据对象：飞书 Base 上下文来源、行动项、决策、风险、飞书文档、飞书任务、飞书日程。
- 验收指标：真实链路验收 17/17 通过；单项工作流覆盖 7 类命令。
- Demo 稳定性：主 Demo 拆成 4 步，分别是项目状态查询、会前卡片、风险周报、推进表对账/文档沉淀，避免一次性长链路超时。
- 效率提升口径：会前资料整理、会后行动项整理、风险周报初稿生成，从人工 30-60 分钟降低到 AI 几分钟生成初稿、人再复核。
- 可推广性：A/B/D 模板不依赖支付项目本身，只要换一套 Base 表和项目资料，就能迁移到研发项目、课程项目、论文项目和活动筹备。

## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

这周主要搞定的是“让主动性 AI 从计划变成产品功能闭环”。

具体环节包括：

- 把主动性 AI 的触发逻辑落到项目指标上：延期、阻塞、缺负责人、重复任务、冲突字段、未闭环决策。
- 把会议生命周期拆成会前、会后、周期复盘和推进表对账，让每个阶段都有主动动作。
- 用真实飞书 Base 做事实库，避免 AI 只基于临时聊天上下文生成内容。
- 用 LLM JSON 让模型输出稳定结构，而不是只写一段自然语言。
- 用 `source_quote` 保留证据，降低 AI 幻觉风险。
- 用 `diff` 机制做安全写入前置层，让 AI 先提出变更建议，再交给人确认。
- 用 OpenClaw 飞书入口承接自然语言触发，让用户可以在飞书群里直接叫小龙虾执行对应工作流。

工具上主要使用了 Python CLI、飞书 Base、lark-cli、OpenClaw、LLM JSON 抽取、run_logs 和 `validate-xiaolongxia` 验收器。

AI 与人的分工也更加清晰：

- AI 负责读上下文、抽取结构、识别风险、生成建议和文档初稿。
- 人负责确认业务事实、处理冲突项、决定是否写回和承担最终责任。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

第一个困难是：主动性 AI 如果设计不好，会变成“AI 自作主张”。例如任务重复时，AI 可能直接合并错；风险负责人缺失时，AI 可能编造负责人；日历授权不足时，AI 可能编造候选时间。

破局方式是建立主动性 AI 的安全边界：

- 可以主动发现，但不能直接越权修改关键数据。
- 可以主动建议，但必须保留来源证据。
- 可以主动指出缺口，但不能编造缺失信息。
- 可以生成 diff，但需要人工确认后再写回。

第二个困难是：会议场景链路很长，包含飞书消息、Base、文档、任务、日历、LLM、日志和验收。如果一次性跑完整链路，演示时间不可控。

破局方式是设计复赛 Demo 的稳定路径：

```text
1. 项目状态查询：证明小龙虾知道真实上下文。
2. 会前卡片：证明它能主动准备会议。
3. 风险周报：证明它能主动发现风险。
4. 推进表对账或文档沉淀：证明它能推动后续闭环。
```

第三个困难是：复赛评分不只看技术，还看实际价值。单纯讲“接了 LLM 和飞书 API”不够有说服力。

破局方式是把每个功能都绑定到真实痛点：

- 会前卡片对应“开会前没人整理背景”。
- 会后抽取对应“行动项散落、责任不清”。
- 风险周报对应“风险发现滞后”。
- 推进表对账对应“任务表和会议事实不同步”。
- 文档沉淀对应“项目知识不能复用”。

## 四、本周随手记（选填）

- 主动性 AI 的关键价值在“提前发现和持续跟进”，不是一次性总结。
- 会议场景最适合作为主动性 AI Demo，因为评委很容易理解会前、会后、周报、待办这些痛点。
- A/B/D 三条链路可以分别打复赛三个维度：A 讲价值，B 讲完整闭环，D 讲技术可控和工程落地。
- 最终演示要把“AI 主动做了什么”说清楚：主动准备、主动抽取、主动巡检、主动对账、主动沉淀。

## 复赛三维度对齐

### 维度 1：完整性与价值（50%）

小龙虾解决的是项目会议协作中的连续痛点：会前信息散、会后行动项漏、风险发现晚、推进表不同步、周报整理慢。AI 在其中承担主动整理和主动提醒的角色，把会议从一次性沟通变成连续项目闭环。

实际价值是：项目经理不再从零整理会前材料、会后纪要和风险周报，而是由 AI 生成结构化初稿，人负责确认和决策。

### 维度 2：创新性（25%）

创新点不是单纯调用大模型，而是把主动性 AI 放进飞书会议流：

- 用真实 Base 做项目长期记忆。
- 用 LLM JSON 做结构化认知。
- 用 A/B/D workflow 做主动行动。
- 用 diff 和 source_quote 做安全边界。
- 用飞书文档、任务、日程和群消息承接结果。

这套方法可以迁移到不同项目，不绑定单个 Demo。

### 维度 3：技术实现性（25%）

技术架构由 5 部分组成：

- 飞书数据层：Base、文档、任务、日程。
- AI 推理层：LLM JSON 抽取和生成。
- 工作流层：A/B/D 三条主动链路。
- 交互层：OpenClaw 飞书机器人入口。
- 验收层：run_logs 和 validate-xiaolongxia。

工程上采用“先生成建议、再人工确认、最后写回”的方式，兼顾主动性和稳定性。
