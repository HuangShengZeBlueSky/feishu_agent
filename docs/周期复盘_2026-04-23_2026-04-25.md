## 一、核心产出（必填）

本周期最满意的产出是：基于“提示词 + skill + 飞书接口 + 小龙虾 + 工作流编排”的第一版可运行方案。

人话解释：用户不需要知道后面有哪些脚本和接口，只要在飞书里 @ 小龙虾并说出需求，小龙虾就根据语义判断要做哪类任务，然后进入对应流程。这个阶段更像是把“项目助手的大脑和入口”搭起来。

执行链路如下：

```text
飞书群消息
  -> OpenClaw Feishu 通道
  -> 小龙虾 skill
  -> 本仓库 Python 入口
  -> A/B/D 路由
  -> 读取真实 Base 上下文
  -> LLM 生成结构化结果
  -> 写文件 / 写 Base / 创建飞书文档 / 群内回复
```

本周期的重点运行例子：

```text
用户在飞书里说：@小龙虾项目上下文管家 帮我整理支付项目会后行动项

系统执行：
1. OpenClaw 收到飞书消息。
2. agent（智能体运行层）选中 xiaolongxia-project 这个 skill。
3. skill 根据提示词和工作流规则，把“会后行动项”识别为会议闭环需求。
4. 本地 Python 逻辑抽取行动项、决策和风险。
5. 输出适合飞书群阅读的回复，并生成本地验证产物。
```

这个周期也明确了一个边界：定时任务在方案上可以作为触发方式，但仓库里当时没有真实 scheduler（定时调度器）实现，所以不把它写成已完成能力；当时主要验证的是自然语言触发和命令触发。

## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

这周主要搞定的是“让飞书里的需求能进入 AI 工作流”。

具体包括：

- 设计了小龙虾的能力边界：把需求收敛到项目上下文、会前准备、会后行动项、风险周报、推进表对账这几类。
- 建立了 `skill`（可复用能力入口）描述，让 OpenClaw agent 能知道什么时候应该调用小龙虾。
- 搭了本地 Python CLI（命令行入口），用于把一个自然语言请求转成可执行工作流。
- 准备了样例输入和本地输出，用来验证行动项、决策、风险等结构化结果是否能稳定生成。
- 写了接口探针和验收脚本，用来检查飞书能力、OpenClaw 通道和本地工作流是否连通。

AI 的分工主要是：帮助整理提示词结构、生成初版流程代码、补齐验收思路和文档；人工主要负责判断边界、确认飞书资源和决定哪些能力先做成固定工作流。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

主要卡点是飞书链路比较长，不是单纯写一个函数就能跑通。消息从飞书到 OpenClaw，再到 skill，再到本地脚本，中间任何一层配置不对都会失败。

破局方式是把问题拆开验收：

- 先验 OpenClaw 是否安装可用。
- 再验 Feishu channel（飞书通道）是否能启动。
- 再验 skill 是否能被选中。
- 最后验本地工作流是否能生成结构化产物。

这样做的好处是，每一层都有单独证据，不会把“模型没回”“飞书没权限”“脚本没跑通”混在一起查。

### 3. 有没有什么这次写得特别顺，或者下次还能直接复用的东西？

可以复用的东西主要有三类：

- `skill` 触发套路：先描述用户会怎么说，再把自然语言映射到固定能力。
- 验收脚本思路：不要只看模型有没有回复，要同时看通道、权限、产物和日志。
- 输出结构：行动项、决策、风险都保留来源证据，后面切到真实数据时可以继续沿用。

最有价值的沉淀是：先用清晰工作流证明“这件事能跑”，再逐步替换底层数据源，而不是一开始就把所有飞书真实资源全部接入。

## 四、本周随手记（选填）

- `skill` 不是魔法，它更像一张“什么时候该调用哪个能力”的说明卡。写清楚触发词、输入、输出和边界，比堆很多泛化提示词更重要。
- 飞书链路调试要分层看：OpenClaw 通不通、飞书账号在不在线、群是否在白名单、模型是否可用、本地脚本是否能跑。
- 定时触发可以作为未来扩展，但在没有真实 scheduler 前，文档里要明确写成“可扩展方向”，不能写成已完成能力。
